What is Kubernetes?

Open source container orchestration platform

Initially built inside Google based on Borg and Omega - Google's internal systems that power search, video and advertising services

Designed as a collection of loosely-coupled components to deploy, scale and maintain containers

Why Kubernetes?

Wide industry adoption: Comcast, IBM, Ebay, Phillips

Huge Community: 1800+ contributors and 40k+ stars on GitHub

Use a single API to manage your infrastructure regardless of physical location (on-prem, co-located, cloud provider)

Kubernetes Use Cases

  • Make sure an application is always available
  • Simultaneously run multiple versions of an application and multiple copies of each version
  • Scale an application automatically based on load
Cluster admin can tell k8s to run N replicas behind a load balancer. Each replica can be geographically distributed for maximum availability. Perform QA on new versions of an application while the previous version is running and serving clients. Kubernetes can automatically scale an application based on CPU usage or the amount of traffic it receives.

Building Blocks: Containers

Containers are a great way to package and run applications.
  • Contains all libraries and code needed by application
  • Less overhead and lower start-up time than VMs
  • Easy to compose and replace

Building Blocks: Deployment

Once we have our application packaged up into a container, how do we run it?
  • Option 1: Manual - Humans
  • Option 2: Automated - Scripts
  • Option 3: Automated - Orchestration Tool

Container Deployment:
Humans

SSH into machines and run containers manually using Docker.

Pros: simple - minimal tooling, configuration and setup required

Cons: not automated, error-prone, doesn't scale, problems require manual troubleshooting

Container Deployment:
Scripts

Custom scripts or tools like Puppet & Chef to run containers on multiple machines
Pros: integrates with existing infrastructure, reproducible, auditable
Cons: manual placement of containers on machines, still doesn't scale well, not easy to move to new infrastructure

Container Deployment:
Orchestration

Orchestrator takes care of running containers on machines which have spare capacity
Pros: automated, self-healing, scalable, portable
Cons: some overhead, learning curve, new tooling

Building Blocks: Orchestrators

Kubernetes, DC/OS, Nomad, etc

Scheduling: match containers to machines

  • by resource needs (CPU, Memory)
  • by affinity requirements (put X near Y)
  • by labels (put X on a "test" machine)

Replication: run N copies

Recovery: handle machine failures

Discovery: find service X
